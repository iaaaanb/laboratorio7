def get_tokens(texto):
    # Busca secuencias alfanuméricas completas.
    patron = r"\b\w+\b"
    tokens = re.findall(patron, texto)
    return tokens

  1. Explique el razonamiento detrás del tokenizador. ¿Qué significa la expresión regular r"\b\w+\b"?
  2. ¿Qué problemas puede tener este tokenizador al trabajar con reviews reales?
  3. ¿Qué otras reglas se podrían añadir para sentiment analysis?

Respuesta:

  1. Es lo que separa las palabras de un texto. La regex r"\b\w+\b" funciona así: \w es un "carácter de palabra" (letras, números y guion bajo), + significa "uno o más seguidos", y los \b marcan los bordes de la palabra (donde empieza y termina). Entonces busca secuencias de letras/números y deja afuera los espacios y signos de puntuación.
  2. Toma las mayúsculas y minúsculas como palabras distintas. No cuenta caracteres como los signos de exclamación, que pueden dar más intensidad. Y lo más grave para sentiment analysis es que, por ejemplo "don't" lo separa en "don" y "t", así que se pierde la negación.
  3. Se podría añadir un filtro que pase todo a minúsculas, que tome en cuenta caracteres como signos de expresión, emojis o guiones, y que expanda las contracciones ("don't" -> "do not") para no perder los "not".

Pregunta 2: Preprocesamiento, stemming y stopwords¶

Las stopwords son palabras frecuentes que suelen aportar poca información semántica. En sentiment analysis hay que tener cuidado: palabras como not, no o never pueden cambiar completamente la polaridad de una review, por lo que no las eliminaremos.

Implementaremos:

  • normalize_text(): normaliza texto y elimina caracteres no alfabéticos.
  • pre_processing(): elimina stopwords y aplica stemming simple.
  • tokenizer(): combina tokenización y preprocesamiento.

STOPWORDS = {
    "a", "an", "the", "and", "or", "but", "if", "to", "of", "in", "on", "for",
    "with", "as", "by", "at", "from", "this", "that", "these", "those", "it",
    "is", "are", "was", "were", "be", "been", "being", "i", "you", "he", "she",
    "we", "they", "my", "your", "our", "their", "me", "him", "her", "them",
}


def normalize_text(text):
    text = text.lower()
    text = re.sub(r"[^a-z\s]", " ", text)
    text = re.sub(r"\s+", " ", text).strip()
    return text


def simple_stem(word):
    substitutions = [
        ("sses", "ss"),
        ("ies", "i"),
        ("ing", ""),
        ("ed", ""),
        ("ly", ""),
        ("s", ""),
    ]
    for old, new in substitutions:
        if word.endswith(old) and len(word) > len(old) + 2:
            return word[:-len(old)] + new
    return word


def pre_processing(vocabulario):
    processed = []
    for token in vocabulario:
        token = token.lower()
        if token in STOPWORDS:
            continue
        processed.append(simple_stem(token))
    return processed


def tokenizer(text):
    normalized = normalize_text(text)
    tokens = get_tokens(normalized)
    return pre_processing(tokens)

Pregunta 3: Bag of Words¶

La representación Bag of Words transforma cada documento en un vector de frecuencias. Cada columna representa una palabra del vocabulario y cada fila representa una review.

Primero implementaremos una versión manual para entender la representación. Luego usaremos CountVectorizer, que es la herramienta habitual para entrenar modelos con scikit-learn.

def build_vocabulary(tokenized_docs, min_df=1, max_features=None):
    document_frequency = Counter()
    for tokens in tokenized_docs:
        document_frequency.update(set(tokens))

    terms = [term for term, df in document_frequency.items() if df >= min_df]
    terms = sorted(terms, key=lambda term: (-document_frequency[term], term))

    if max_features is not None:
        terms = terms[:max_features]

    return terms


def bag_of_words(tokenized_docs, vocabulary):
    rows = []
    for tokens in tokenized_docs:
        counts = Counter(tokens)
        rows.append({term: counts.get(term, 0) for term in vocabulary})
    return pd.DataFrame(rows, columns=vocabulary)

def show_top_words(model, feature_names, n=10):
    if hasattr(model, "coef_"):
        weights = model.coef_[0]
        top_positive = pd.DataFrame({
            "palabra": feature_names[np.argsort(weights)[-n:][::-1]],
            "peso": np.sort(weights)[-n:][::-1],
        })
        top_negative = pd.DataFrame({
            "palabra": feature_names[np.argsort(weights)[:n]],
            "peso": np.sort(weights)[:n],
        })
        return top_positive, top_negative

    if hasattr(model, "feature_log_prob_"):
        # Diferencia entre log-probabilidad positiva y negativa.
        weights = model.feature_log_prob_[1] - model.feature_log_prob_[0]
        top_positive = pd.DataFrame({
            "palabra": feature_names[np.argsort(weights)[-n:][::-1]],
            "peso": np.sort(weights)[-n:][::-1],
        })
        top_negative = pd.DataFrame({
            "palabra": feature_names[np.argsort(weights)[:n]],
            "peso": np.sort(weights)[:n],
        })
        return top_positive, top_negative

    raise ValueError("Modelo no soportado")

    for j, palabra_b in enumerate(palabras):

            continue

Respuesta:

  • Pasa porque los embeddings agrupan las palabras según el contexto en que se usan, no según si son positivas o negativas. Pregunta 9: Entrenamiento del modelo BERT¶

BERT ya fue preentrenado cuando fue diseñado y validado con millones de textos. ¿Por qué entonces necesitamos entrenarlo de nuevo con nuestras reseñas?

Respuesta: Porque una cosa es que BERT entienda el lenguaje y otra es que sepa hacer nuestra tarea. Cuando Google lo preentrenó, BERT aprendió cosas generales del idioma (gramática, contexto, significado de las palabras), pero eso es conocimiento general: no sabe todavía clasificar reviews en positivas o negativas. Para eso necesitamos el "fine-tuning": ajustarlo con nuestras reseñas etiquetadas para que aprenda específicamente esta tarea, porque la capa final que decide positivo/negativo parte de cero. Además nuestras reseñas tienen su propio vocabulario y estilo (de restaurantes, productos, etc.) que es distinto a los textos generales con que se entrenó, así que entrenarlo de nuevo lo adapta a nuestros datos. La gracia es que como ya sabe el idioma, no partimos de cero: nos basta un entrenamiento corto y muchos menos datos que si lo hiciéramos todo desde el principio (esto se llama transfer learning).
